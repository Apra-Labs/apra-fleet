Apra Fleet -- the idea in plain language

A team of AI developers that keeps a shared notebook

Apra Fleet runs your development like a small team: a planner breaks the work down, doers write the code, reviewers check it -- each on an AI model sized to the job.

Until now, every AI tool had one flaw: total amnesia. Finish a task, close the session, and everything it figured out about your codebase is gone.

The new build fixes that with two memories:
1. A map of the code -- automatically built and kept current, so agents navigate by structure instead of searching blindly.
2. A notebook of lessons -- everything the agents figure out while working ("this filter lives in the store layer", "never break X") is written down, fact-checked by a reviewer before it's trusted, and saved inside your git repo -- so the whole team, human and AI, inherits it.

The result: the second time the fleet touches any part of your code, it already knows it.
SAY: "One sentence: it's a team of AI developers that keeps a shared, fact-checked notebook in your repo -- so it gets smarter and cheaper the more it works for you."
How it works -- the knowledge lifecycle

How the fleet stores and reuses knowledge

STORE — during and after every sprint
1. DISCOVER
planner / doer / reviewer hit a durable fact while working (a convention, a constraint, a gotcha)
→
2. kb_capture
saved immediately, clamped to INFERRED -- no agent can self-certify
→
3. REVIEWER
independent verdict on the code that demonstrates the fact
→
4. KB AGENT
kb_promote to CONFIRMED only if the reviewer APPROVED. The curator.
→
5. kb_export
.fleet/kb-canonical.json auto-commits to git -- the shareable bible
RETRIEVE — at the start of the next sprint / session / clone
6. RECALL (kb_session_prime)
planner/doer recall prior learnings for the task's symbols before touching a file
→
7. CONFIRMED hits
prior learnings surface (a cold clone seeds from the git bible)
→
8. REUSE
agent trusts the fact, skips re-reading the code, picks a cheaper model
Trust tiers, enforced in code: UNVERIFIED (raw / harvested) → INFERRED (captured, unreviewed) → CONFIRMED (reviewer-approved). Only promotion mints CONFIRMED; the bible carries only CONFIRMED.
SAY: "This is the whole engine. Agents capture what they learn as they work; a reviewer's approval promotes it to trusted; the KB Agent curates it and commits it to your repo; and the next session recalls and reuses it. Store on the top row, recall on the bottom."
Two kinds of knowledge -- don't confuse them

Code Graph vs Agent Learning

CODE GRAPH — code intelligence (gitnexus)

What the code IS: symbols, call graph, module structure.
Machine-derived, rebuilt on demand, always current, disposable -- delete it and re-index for the identical result.

Tools: code_context, code_impact, code_query, code_graph

Answers: "Where is X? Who calls Y? What breaks if I change Z?"

AGENT LEARNING — Knowledge Bank

What we FIGURED OUT: conventions, constraints, gotchas, decisions -- the things the code does not tell you.
Earned through work, reviewer-gated, persists in git. Precious -- lost forever if not captured.

Tools: kb_capture, kb_promote, kb_session_prime (recall), kb_query

Answers: "Why is it built this way? What must I not break?"
They are complementary, and the current build joins them: code_context on a symbol also inlines any CONFIRMED Knowledge-Bank facts about it -- structure and hard-won wisdom in one call.
SAY: "Two different things. The code graph is the map -- always accurate, rebuildable. Agent learning is the wisdom -- earned, reviewed, and easy to lose. v0.3.4 had neither; the new build has both, and links them."

Under the hood -- for the engineers in the room

It's all MCP tool calls. No magic, no cloud.

STORE -- what actually happens

Agent makes a normal MCP tool call: kb_capture({title, content, symbols, source_files})
-> the local apra-fleet server (a Node process on your machine) writes one row into a per-repo SQLite DB (kb.sqlite) with:
  - an FTS5 full-text index for search
  - a trust column (clamped to INFERRED on capture)
  - file-content hashes of the source files (for staleness)
  - duplicate/contradiction detection (AUDN) at write time
After review, kb_promote flips trust to CONFIRMED and kb_export writes the CONFIRMED set to .fleet/kb-canonical.json and git-commits it.

RETRIEVE -- what actually happens

One tool call at session start: kb_session_prime({hint_symbols, hint_modules})
-> the server runs a pipeline and returns ranked entries:
  1. FTS search (OR across hint terms) over the SQLite DB
  2. staleness check -- re-hash source files; drop entries whose code changed
  3. graph expansion -- also recall facts about callers/callees of the hinted symbols
  4. cold-seed -- fresh clone? load from the git bible
Mid-task: kb_query before reading an unfamiliar file.
No embeddings service, no vector database, no external calls. SQLite + FTS5 + file hashes + a JSON file in git -- boring on purpose, so it's fast, inspectable, and yours.
SAY: "When someone asks 'how is it stored' -- it's a tool call into a local SQLite DB with full-text search, trust columns, and file hashes; the confirmed subset is a JSON file committed to git. Retrieval is one tool call that searches, staleness-checks, and ranks."

Apra Fleet -- part 2: the experiment

The Amnesia Test

Every AI coding tool is brilliant and forgets everything the moment a session ends. It re-reads the same files, re-derives the same facts, and repeats the same mistakes -- every single time.

We ran the same two tasks on the same codebase twice -- once on our last public release, once on the new build -- to show, concretely, what changes when a fleet actually remembers.
SAY: "This isn't a slideshow of features. It's a real head-to-head -- identical repo, identical work, same machine. The only variable is the build."

SAY: "Same repo, same prompts, same orchestrator. One build remembers what it learns; the other doesn't. Watch what that buys."
The tasks (identical for both environments)

Two sprints, back to back

  • Sprint 1 -- Note archiving: add an archived flag to notes, add archive / unarchive endpoints, and exclude archived notes from the default listing.
  • Sprint 2 -- Pagination: paginate GET /api/notes into a { data, total, page, limit } envelope -- and it must compose correctly with the archived filter sprint 1 just built.
Sprint 2 is the trap. It reaches straight back into the same listing code sprint 1 changed. A fleet with memory recalls what it just did; a fleet without memory re-learns it from a cold start.
SAY: "The second sprint deliberately depends on the first. That dependency is exactly where memory pays off -- or where its absence hurts."
How one sprint runs -- the pipeline

Four roles, each on a right-sized model

PLANNER
Reads the goal, builds a task DAG, assigns each task a model tier. Strongest model.
→
PLAN-REVIEWER
Adversarially reviews the plan. Loops to APPROVED before any code.
→
DOER
Implements one task at a time, commits each, stops at VERIFY.
→
REVIEWER
Independently re-runs build+tests, verdict APPROVED / CHANGES NEEDED.
→
KB AGENT
Curates what was learned, promotes confirmed facts, commits the bible. Env B only.
Both builds run planner -> plan-reviewer -> doer -> reviewer. The current build adds the KB Agent at the end of every cycle -- the step that turns a finished sprint into retained, reusable knowledge.

The honest part -- say this out loud

What this doesn't claim

We are not showing a "40% fewer tokens" headline. The two builds structure their work differently (Env A used a separate worktree per sprint; Env B layered both on one branch), and the sandbox cost telemetry didn't populate cleanly for either side -- so a raw token or commit-count number would not be defensible.

The verified, defensible claim is the capability: real reuse happened in Env B (8 retrievals, priming on prior work) and is structurally impossible in Env A. And v0.3.4 can't even measure its own tokens -- its logs read Act tokens: 0.
SAY: "Credibility beats a big number. I'm deliberately not cherry-picking a benchmark. The win is capability, and it's real."
The twist -- and the best part of the story

The demo caught a release-blocker

The first Env B runs showed no knowledge activity at all. Root cause: the shipped agent definitions hard-coded tool lists that excluded every KB and code-intelligence tool -- so the standard flow's planner, doer, and reviewer literally could not call them. The entire knowledge layer was inert in the real product path, working only when orchestration was hand-driven.

We found it, fixed it (grant the agents MCP access), and re-ran clean -- every number in this deck is from the fixed run.
A rigorous head-to-head that catches your own release-blocker before it ships is worth more than any favorable benchmark.
SAY: "This is my favorite slide. We didn't just measure the feature -- our own test caught it silently broken in the release path, and we fixed it in the same session."
Takeaway

A fleet that gets cheaper, more consistent, and more aligned the more it works

Knowledge captured once is retrieved forever. A correction made once propagates forever. A standing instruction given once applies forever -- and all of it lives in your repository, committed and diffable, not in a vector database you rent.

v0.3.4 finished each sprint and forgot everything. The new build learned from it -- and proved it by catching its own bug.
SAY: "That's the pitch. Same work gets done, but the fleet compounds -- and the knowledge is yours, in your repo."
