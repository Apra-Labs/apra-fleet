Upgrade Head-to-Head: released v0.3.4 vs current build

Same NoteAPI repo, same two sprints (note archiving, then pagination), same machine. Env A = the last public release; Env B = the branch under review. Run 2026-07-13. Every number below was verified directly against the sandboxes.

Env A -- apra-fleet v0.3.4 (released)

Installed from the GitHub release asset. 10-step install, no Knowledge Bank, no code-intelligence layer. Both features delivered and tested (build+lint+test green), driven entirely by beads + file reads.

Env B -- current build

Installed from the branch. 11-step install -- the extra step is "Setting up Knowledge Bank and code intelligence." Same two features delivered, with the KB + gitnexus layer active and isolated to a per-env data dir.

THE FINDING THAT MATTERS MOST

P0 BUG CAUGHT BY THIS DEMO (yashr-9y5, now fixed) The first Env B attempts showed no KB activity. Root cause: the shipped planner/doer/reviewer agent definitions hard-coded tool allowlists that excluded every MCP tool, so the standard /pm flow's subagents physically could not call kb_session_prime, kb_capture, or code_*. The entire knowledge layer was inert in the real product flow and only ever worked when orchestration was hand-driven with unrestricted agents. The fix -- grant mcp__apra-fleet__* to the three agents -- is on submodule branch fix/agent-mcp-tool-grants; after it, a fresh Env B run captured all 16 entries above. A rigorous head-to-head that catches your own release-blocker before ship is worth more than a favorable benchmark.

BOTH BUILDS DELIVERED THE FEATURES

To be fair to v0.3.4: it is a capable release. Both environments implemented note archiving and pagination correctly, with full doer/reviewer loops and passing test suites. The difference is not whether the work got done -- it is that the current build learned from doing it, kept that knowledge, and made it reusable and shareable. v0.3.4 finished each sprint and forgot everything.
