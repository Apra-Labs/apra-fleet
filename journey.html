1
One-Time Install
once per machine

Everything starts with apra-fleet install run from inside a git repo. This wires the MCP server into Claude Code globally and deploys all skills.

User $ apra-fleet install
# or: node dist/index.js install (dev mode)

Step 1-6: Binary, hooks, statusline, MCP registration
-> Writes MCP entry into Claude Code settings (global, not per-project)
-> Installs git hooks: post-commit triggers kb_invalidate (marks changed files stale in KB)

Step 7: Install PM skill
-> Copies skills from vendor/apra-pm OR dist/skills/pm
-> THEN overlays skills/pm/ from repo root (apra-fleet-owned overrides)
-> Destination: ~/.claude/skills/pm/ (SKILL.md, init.md, index.md, tpl-*.md, ...)

Step 8: Install fleet skill
-> Copies skills/fleet/ -> ~/.claude/skills/fleet/

Step 8b: Install Beads (bd CLI)
-> npm install -g @beads/bd (idempotent)

Step 9: KB + code intelligence setup
-> Removes legacy gitnexus entry from .mcp.json (cleanup)
-> Writes ~/.apra-fleet/data/code-intelligence/config.json {provider: 'gitnexus'}
-> Appends to ~/.claude/CLAUDE.md (sentinel-guarded, idempotent):
"When code_graph/impact/query/context tools are available, use them for
symbol lookups -- never use grep or file reads for structural questions"

Result
-> Fleet MCP server runs globally, all 12 tools available in every Claude Code session
-> /pm and fleet skills installed and ready
-> Claude Code is instructed to prefer fleet code intelligence tools automatically

3
/pm init <project>
once per project
User types /pm init streamsurv

PM Skill (reads init.md)
-> Determines PM context file name from llmProvider (CLAUDE.md / GEMINI.md / AGENTS.md)
-> Creates ~/pm/streamsurv/ folder
-> Writes from templates: status.md, requirements.md, design.md, deploy.md, permissions.json
-> Adds row to projects.md

Step 5: Code intelligence index
execute_command on any available member -> repo root
npx gitnexus analyze (timeout 300s, non-fatal)
# Indexes codebase so code_graph/impact/query/context tools work
# Now delegated to /pm index if not run at init

Step 6: Beads init
cd ~/pm && bd init (idempotent, one central DB for all projects)
bd create "sprint: streamsurv" --type epic -> epic-id saved in status.md
PM root is one folder: ~/pm/ contains one subfolder per project. Beads DB is in ~/pm/ (not inside each project repo). All sprint artifacts (PLAN.md, progress.json, feedback.md) live in ~/pm/<project>/ -- never inside the git repo until the member sends them there.
4
/pm plan <requirement>
start of each sprint
User provides requirements.md (written by PM or user)

PM Skill (reads single-pair-sprint.md, plan-prompt.md)

Phase 2a: Generate plan
send_files -> doer: requirements.md + tpl-plan.md
execute_prompt -> doer (resume=false)
# Doer reads files, produces PLAN.md with phase-ordered tasks + tier per task
receive_files <- doer: PLAN.md

Phase 2b: Plan review loop
send_files -> reviewer: PLAN.md + requirements.md + tpl-reviewer-plan.md
execute_prompt -> reviewer (model=premium, resume=false)
# Reviewer checks: tasks cover requirements, risks front-loaded, tiers correct
receive_files <- reviewer: feedback.md
-> APPROVED: save planned.json (immutable copy), push Beads tasks
-> CHANGES NEEDED: send feedback to doer, doer revises PLAN.md, re-review

After plan approval
bd create "T1.1: <title>" -p 1 --parent <epic> --assignee doer (one per task)
bd dep add <T1.2-id> <T1.1-id> (wire dependencies from PLAN.md)
5
/pm start <member>
each execution phase
PM Skill (reads doer-reviewer.md)

Setup (doer-reviewer.md checklist)
1. update_member(doer, icon=circle, unattended='auto')
2. update_member(reviewer, icon=square, same color)
compose_permissions -> doer (role=doer, project_folder)
# Detects stacks (node/python/rust), writes .claude/settings.local.json

Pre-flight checks
fleet_status -> confirm doer is idle
execute_command -> doer: git status && git branch --show-current
# Confirm clean tree, correct branch

Resolve next task
nextTask = planned.json.tasks.find(t => t.status === "pending")
tier = nextTask.tier (cheap / standard / premium)
resume = (nextTask.phase === lastDispatchedPhase) from status.md

Send task harness
send_files -> doer:
CLAUDE.md (filled tpl-doer.md: {{PROJECT_NAME}}, {{branch}}, {{base_branch}})
PLAN.md, progress.json, requirements.md, design.md

execute_command -> doer: echo 'CLAUDE.md' >> .gitignore
# Ensure context file is never committed

bd update <task-id> --status in_progress

Dispatch (always in background Agent)
execute_prompt member=doer, model=<tier>, resume=<bool>, timeout_s=600
# PM rule 8: club all fleet calls into ONE background Agent -- never call execute_prompt
# directly from PM context; always wrap in Agent(run_in_background=true)
6
Inside a Member Session
every execute_prompt call

This is what happens inside the fleet server and on the member machine when execute_prompt fires.

Fleet Server (execute-prompt.ts)

1. Resolve member from registry
2. ensureCloudReady() -- start EC2 instance if stopped (cloud members only)
3. Guard: reject if member already inFlight (one-at-a-time)
4. getProvider(agent.llmProvider) -- claude / gemini / opencode / ...
5. Resolve model:
modelTiers (free-form) -> modelCheap/Standard/Premium (legacy enum)
-> user global override -> provider defaults
6. Build auth env prefix (API keys, SSH agent forwarding)
7. Mint session UUID (or reuse existing sessionId for resume)
8. writePromptFile: write prompt to .fleet-task.md in member workFolder
9. stallDetector.add: background monitor (clears busy state if stalled)
10. Build CLI command, e.g.:
claude -p --session <uuid> --model <model> --dangerously-skip-permissions
--output-format stream-json --max-turns 50
11. execCommand -- run via SSH (remote) or direct (local)
12. Stale session retry if exit != 0 and had a prior session
13. Server overload retry (wait 5s, fresh session)
14. parseResponse -- extract text, sessionId, token usage
15. touchAgent -- persist sessionId for next resume
16. *** AUTO-HARVEST (fire-and-forget) ***
kb_harvest({ session_transcript: parsed.result }) -- extracts learnings from output
Runs AFTER session ends, on PM side. Member never sees it.
17. deletePromptFile -- clean up .fleet-task.md
18. Return response text to PM

Member Machine (Claude Code CLI, running in member workFolder)

-> Reads .fleet-task.md (the prompt)
-> Reads CLAUDE.md (context file -- role, branch, rules, KB instructions)
-> Reads progress.json (find next pending task)
-> Reads PLAN.md (task details)
kb_session_prime -- prime KB context (see section 9)
-> Dispatches recommended code intelligence calls
-> Executes task (Edit / Write / Bash / Read / Grep)
kb_capture -- after key decisions/discoveries
-> Commits: ONE task at a time, then tests, then commit
-> Updates progress.json (task -> "completed")
-> Continues to next pending task
-> At VERIFY checkpoint: git push, report status, STOP
kb_harvest -- at end of session (explicit, inside session)
7
VERIFY Checkpoint -- Doer-Reviewer Loop
at every verify task in PLAN.md
PM detects VERIFY task in progress.json

Sync reviewer to doer HEAD
execute_command -> reviewer:
git fetch origin && git checkout <branch> && git reset --hard origin/<branch>
execute_command -> reviewer: git rev-parse HEAD (must match doer's pushed HEAD)

Prepare reviewer
compose_permissions -> reviewer (role=reviewer)
send_files -> reviewer: CLAUDE.md (filled tpl-reviewer.md)
bd update <verify-id> --status in_progress

Dispatch reviewer
execute_prompt reviewer, model=premium, resume=false

Reviewer runs (inside session)
git log --oneline <base>..<branch> -- scope the diff
Read progress.json, PLAN.md, requirements.md
git diff -- check all changed files against sprint requirements
npm run build / npm test -- must all pass
Verify each task against done criteria in PLAN.md
Check file hygiene (no temp files, no unrelated changes, CLAUDE.md not committed)
Writes feedback.md (APPROVED or CHANGES NEEDED)
git commit feedback.md && git push

PM reads verdict
APPROVED ->
bd close <verify-id>
Proceed to next phase (or sprint completion)
CHANGES NEEDED ->
bd create "<finding>" -p 0 --parent <epic> (one per HIGH finding)
execute_prompt -> doer (resume=true): fix feedback
Re-dispatch reviewer (resume=true) to re-check
# Safeguard: 3 cycles max; if still unresolved after 3, escalate to user
Dispatch Conditionresume value
New phase (phase number changed)false
Same phase, continuing taskstrue
Reviewer CHANGES NEEDED -> doer fixtrue
Role switch (doer <-> reviewer)false
After stop_prompt cancellationfalse
Initial review dispatchfalse
Re-review after doer fixestrue
8
Code Intelligence Tool Call Chain
any time a member calls code_graph / code_impact / code_query / code_context
Member calls code_graph({ symbol: "handleIPChange", repo: "/path/to/repo" })

Fleet MCP Server (src/tools/code-intelligence.ts)
-> getProvider()
reads ~/.apra-fleet/data/code-intelligence/config.json
{ provider: 'gitnexus' } (default)
PROVIDERS['gitnexus'] -> GitNexusProvider instance

GitNexusProvider (src/tools/code-intelligence-gitnexus.ts)
-> getGitNexusClient() (singleton MCP client)
First call: spawns npx -y gitnexus mcp (StdioClientTransport)
Subsequent calls: reuses the same process
-> client.callTool({ name: 'call_graph', arguments: { symbol, repo } })
# Fleet tool names -> gitnexus tool names:
code_graph -> call_graph
code_impact -> impact
code_query -> query
code_context -> context

gitnexus MCP server (npx gitnexus mcp, local process)
-> Queries its local index (built by npx gitnexus analyze)
-> Returns call graph / impact / search results

Member receives the structured result and uses it for the task
Adding a new provider: Create one file implementing the CodeIntelligenceProvider interface (4 methods: graph, impact, query, context) and add one entry to the PROVIDERS map. Nothing else changes -- fleet routing, tool schemas, member templates all stay the same.
9
Knowledge Bank Flow
across every session
SESSION START -- Prime
Member calls kb_session_prime({ session_files, hint_symbols, hint_modules })
-> SQLiteProvider.prime()
Checks freshness of session_files against KB hashes
Fetches top KB entries (L1 FTS on title+summary)
Also queries global KB (top 3 matching entries merged in)
Builds recommended_code_calls:
{ tool: 'code_context', args: { name: symbol } } for each hint_symbol
{ tool: 'code_impact', args: { target: file, direction: 'upstream' } } for each file
Returns: { session_warm, stale_files, top_entries, fresh_summaries, recommended_code_calls, token_estimate }

Member
-> Re-reads stale files (their KB cache is outdated)
-> Dispatches each recommended_code_calls entry using fleet code intel tools

DURING SESSION -- Capture
Member calls kb_capture({ type, title, summary, content, symbols, source_files })
-> AUDN deduplication runs:
add (new): no match found
none (duplicate): skip, return existing id
update (refines): link new entry to old by default -- both stay live; pass supersedes to retire the old one for real
flagged (contradiction): flag for review, keep both with reference
-> Stored in ~/.apra-fleet/data/knowledge/<repo-slug>/kb.sqlite

SESSION END -- Harvest (two paths)
Member calls kb_harvest({ session_transcript }) -- explicit, inside session
-> Scans transcript for learnings
-> Calls kb_capture for each finding (AUDN dedup applies)
-> Returns { entries_captured, entries_updated, entries_skipped }

Fleet Server auto-calls kb_harvest fire-and-forget -- AFTER session ends
kb_harvest({ session_transcript: full_output, session_id }) from execute-prompt.ts:323
Runs on PM side. Member never sees it. Belt-and-suspenders on top of explicit harvest.

GIT HOOK -- Auto-invalidate
Post-commit hook (installed by apra-fleet install)
-> Runs on every commit in project repos
kb_invalidate({ files: [changed files] })
-> Marks those files stale in the KB
-> Next kb_session_prime will include them in stale_files -> member re-reads them
KB Storage
Project KB: ~/.apra-fleet/data/knowledge/<slug>/kb.sqlite
Global KB: ~/.apra-fleet/data/knowledge/global/kb.sqlite

Slug derived from git remote URL. Two repos on the same machine get separate KBs.
KB Entry Types
context-cache file summaries (hash-checked for freshness)
learning session insights
knowledge codebase facts
runbook procedures

Confidence: UNVERIFIED -> INFERRED -> CONFIRMED (via kb_promote)
10
/pm index <project>
after init, large merges, or when code intel feels stale
User types /pm index streamsurv

PM Skill (reads index.md)
1. Read ~/pm/streamsurv/status.md -- find repo path + available member
2. Pick member whose work_folder is the project repo
If none, use any local online member, set run_from to repo path
execute_command
run_from: /path/to/repo
command: npx gitnexus analyze
timeout_s: 300
Non-fatal: if fails (no package.json, bad language), log and continue

gitnexus analyze
-> Parses source tree, builds call graph, writes local index on disk
-> Subsequent calls are incremental (only changed files re-parsed)
-> Index used by GitNexusProvider for all 4 fleet code intel tools
11
/pm cleanup + PR
after all phases APPROVED, when user says "raise a PR"
PM Skill (reads cleanup.md)

Cleanup on both doer and reviewer
execute_command on each member:
git rm --cached .fleet-task*.md (remove ephemeral prompt files)
git rm PLAN.md progress.json feedback.md requirements.md design.md
For CLAUDE.md/GEMINI.md/AGENTS.md: restore from origin/main if it existed there
(preserves project deliverable; only deletes if it was a pure sprint artifact)
git commit -m "cleanup: remove fleet control files" && git push

Close Beads epic
bd close <epic-id>

Raise PR (PM runs gh directly -- never delegates to member)
gh pr create --title "..." --body "..."
gh pr checks (wait for CI)
bd note <epic-id> "PR: <url>"

STOP
Surface PR URL + CI status to user
Do NOT merge. Merge is always the user's decision.
