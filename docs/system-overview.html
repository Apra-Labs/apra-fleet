Apra Fleet -- Complete System Overview

How two knowledge types flow, how agents communicate, and where everything is stored.

Two types of knowledge

Knowledge Bank (KB)
Semantic, behavioral, durable -- survives across sprints
What it stores
knowledge runbook context-cache learning
Engine
SQLite + FTS5
per-repo scoped
Examples
"jitter added AFTER maxDelayMs cap"
"shouldRetry receives 1-based attempt"
"AUDN requires symbol AND file overlap"
Code Memory (GitNexus)
Structural, call-graph, symbol index -- derived from AST
What it stores
call graphs symbol defs imports impact maps
Engine
AST analysis
gitnexus index
Examples
"SqliteProvider.query() calls filterByType()"
"Changing kb_capture.ts affects: index.ts, 3 tests"
"retry() is exported from index.ts"
Rule: Never use grep or file reads for structural questions (who calls what, what imports what) -- use code_graph, code_impact, code_query, code_context. Never use these tools for behavioral facts -- use kb_query.

KB -- How it gets written, read, and maintained

Write paths
KB Agent calls kb_capture after every reviewer verdict
KB Agent calls kb_promote to raise confidence on existing entries
Doer/Reviewer call kb_harvest at end of session -- regex scan of session output, feeds KB Agent
kb_setup installs a git post-commit hook that calls kb_invalidate on changed files automatically
Read paths
Session start: kb_session_prime -- returns top entries, stale files, recommended code calls
Retrieve-first: before reading any unfamiliar file/function, doer calls kb_query -- if CONFIRMED/INFERRED entry exists, trust it, skip source read
KB Agent calls kb_query before EVERY capture -- dedup check
kb_context -- checks if a file's summary is fresh (git-hash bound)
Maintenance
AUDN: every capture checks for near-duplicates (symbol + file overlap + FTS title match)
Confidence decay: INFERRED entries with no source_files, not accessed in 30 days -> demoted to UNVERIFIED at next prime()
Git hook: file change -> kb_invalidate marks entries for that file as stale automatically
/pm kb-review: surfaces flagged contradiction pairs for human resolution

Confidence ladder

LevelMeaningHow it's reachedHow it decays
CONFIRMED Reviewer APPROVED the code this entry describes. Highest trust. KB Agent calls kb_promote after APPROVED verdict, or kb_capture with confidence=CONFIRMED Only via contradiction flag or manual invalidation
INFERRED KB Agent verified by reading source. Trusted but not reviewer-stamped. kb_capture(confidence='INFERRED') or promoted from UNVERIFIED If no source_files, not accessed/promoted in 30+ days -> auto-demoted to UNVERIFIED at next prime()
UNVERIFIED Transcript-only. Low trust, useful as a hint. kb_harvest auto-scan, or KB Agent with CHANGES NEEDED verdict Lowest level -- no further decay

Code Memory (GitNexus) -- How it gets built and read

Build / Update
PM runs /pm index -> npx gitnexus analyze on the project repo via a fleet member
Run once after /pm init, again after large merges
Builds AST call-graph, symbol index, import maps -> stored in .gitnexus/
Read (4 MCP tools)
code_graph -- who calls this function, who is called by it
code_impact -- if I change X, what else breaks?
code_query -- where is symbol Y defined?
code_context -- full symbol context with callers/callees
Freshness
Index becomes stale after large refactors or merges
No automatic invalidation -- PM re-indexes manually
Does NOT decay on its own -- structural facts stay valid until the code changes
Code Memory answers STRUCTURAL questions ("what calls what"). KB answers BEHAVIORAL questions ("what does this do, what are the non-obvious constraints"). Both are needed -- neither replaces the other.

Agents and their roles

PM Orchestrator
this session never writes code
Reads state from git + beads. Dispatches all other agents. Manages worktrees, permissions, git plumbing. Raises PRs. Loops until APPROVED.
KB: reads status.md only. No KB calls directly.
Planner
local subagent strongest model
Reads requirements.md (+ design.md). Writes PLAN.md with phase-ordered tasks, each with an assigned model. Self-critiques. Commits.
KB: none -- plan is derived from requirements, not KB.
Plan-Reviewer
local subagent strongest model
Reads requirements.md + PLAN.md. Writes feedback.md (APPROVED / CHANGES NEEDED). Gates plan execution.
KB: none.
Doer
fleet member plan-assigned model
Implements tasks in order. Commits after each. Stops at VERIFY. Runs lint + tests. Pushes to origin.
KB: kb_session_prime (start) -> retrieve-first kb_query -> kb_harvest (end). NO kb_capture.
Reviewer
fleet member strongest model
Reviews diff against plan + requirements. Runs lint + tests. Writes feedback.md verdict. Commits + pushes.
KB: kb_session_prime (start) -> kb_harvest (end). Verdict (APPROVED / CHANGES NEEDED) directly sets KB confidence ceiling.
KB Agent
fleet member (doer's machine) cheap model
Runs after EVERY reviewer verdict. Reads diff + session output + feedback.md. Captures durable knowledge. No code writes, no git commits.
KB: kb_query (dedup) -> kb_capture -> kb_promote -> report inline. ALL writes go through KB Agent.

Where everything is stored

Knowledge Bank (KB)
~/.apra-fleet/data/knowledge/<repo-hash>.db
SQLite with FTS5 full-text index
One DB per git repo (scoped by repo root hash)
Tables: entries, links, sessions
Entries have: type, confidence, content, source_files, symbols, content_hash
Code Memory (GitNexus)
<project-repo>/.gitnexus/
AST-derived call-graph index
Built by npx gitnexus analyze
Queried via 4 MCP tools
Must be re-indexed after large merges
Beads (Task DB)
~/pm/.beads/*.db
Dolt-backed SQLite (git-versioned rows)
One DB for ALL projects and sprints
Tracks: epics, tasks, status, assignees, findings
PM root, not inside project repos
Sprint State
<project-repo>/<branch>/
PLAN.md -- task list with models
progress.json -- live execution state
feedback.md -- latest review verdict
requirements.md -- intent document
Committed to feature branch, pushed to origin
Fleet Config
~/.apra-fleet/
members.json -- registered members
data/knowledge/config.json -- KB provider config
hooks/ -- post-register and other hooks
credentials/ -- encrypted credential store
PM Status
~/pm/<project>/status.md
Phase log, member assignments, mode
KB Agent run summaries
Not committed to git -- PM-local only
Updated after every dispatch completes

Key invariants

RuleWhy
Only KB Agent calls kb_capture / kb_promote Doer and reviewer are too embedded in implementation to evaluate knowledge quality. KB Agent reads diff + verdict + transcript with fresh context.
CONFIRMED requires reviewer APPROVED verdict Reviewer is the quality gate. CONFIRMED entries are trusted by future doers without source reads. Must be earned.
git is the single source of truth for sprint state PM, doer, reviewer all share state through committed files on the feature branch. Memory is not trusted across sessions.
Doer uses retrieve-first before reading source Avoids re-reading files that prior agents have already summarized. CONFIRMED/INFERRED entries are faster and sufficient.
KB is per-repo scoped apra-sandbox KB and apra-fleet KB are separate. Learnings stay relevant to their codebase. PM queries the wrong KB if not running from the right repo.
KB Agent runs on doer member, cheap model, fresh session Doer already has the correct branch checked out. Cheap model is sufficient for evaluation. Fresh session avoids inheriting doer's biases.
