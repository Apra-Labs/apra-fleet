# Hub-Spoke Wire Protocol Design

Status: design document, 2026-07-04. Answers docs/hub-spoke-master-plan.md
section 11, beads item 1 (apra-fleet-us9.1). Scope: the outbound channel each
machine's apra-fleet.exe (tier 2, the spoke) maintains to fleet.apralabs.com
(tier 3, the hub). Does NOT cover the tier-1 <-> tier-2 localhost MCP transport
(that is `src/services/http-transport.ts`, unchanged by this document) or the
persistence backend (that is the ADR, apra-fleet-us9.3).

## 1. Design constraints (from the master plan)

- Outbound-only from the spoke (works behind NAT/firewalls). The hub never
  initiates a connection to a spoke; it only ever writes onto a channel the
  spoke opened.
- One long-lived channel per spoke (per machine JWT), not per member. A
  machine may host multiple members; all of their hub-bound traffic
  multiplexes over the one spoke channel, demultiplexed by member_id inside
  the envelope.
- workspace_id is the hard boundary, enforced at the hub on every envelope,
  both on admission (from a spoke) and on delivery (to a spoke).
- At-least-once delivery for message relay (send_message, execute_command,
  execute_prompt routing). Plain pub/sub is disqualified: it silently drops
  messages to a briefly-disconnected spoke, which is unacceptable for
  execute_command (master plan section 6).
- Presence is volatile and best-effort; reconnect + re-announce is the
  expected recovery path, not a failure mode to design around at all costs.
- Self-host parity matters: the protocol must not assume a specific
  persistence backend (Redis Streams, Postgres LISTEN/NOTIFY, NATS
  JetStream are all legitimate implementations of the same wire contract).
  This document specifies the WIRE contract; the ADR (us9.3) picks the
  server-side implementation.

## 2. Transport

- HTTPS with a long-lived Server-Sent-Events (SSE) stream, hub -> spoke, for
  delivery, mirroring the existing tier-1<->tier-2 transport shape
  (`src/services/http-transport.ts`) so the same client-side SSE handling
  code can largely be reused/adapted for tier 2 <-> tier 3.
- HTTPS POST, spoke -> hub, for everything the spoke originates (envelope
  submission, heartbeats, presence announce/renew). Request/response, not
  push -- this keeps the spoke's outbound path a normal HTTP client, which
  works through corporate proxies more reliably than a bidirectional
  WebSocket would.
- Every request and every SSE reconnect carries `Authorization: Bearer
  <machine JWT>`. The JWT's workspace_id claim is authoritative for scoping;
  the hub never trusts a workspace_id/member_id supplied only in the
  envelope body without cross-checking it against the JWT.
- Reconnect/backoff: exponential backoff with jitter, base 1s, cap 60s,
  starting immediately on stream drop. No backoff reset until a
  successfully-processed heartbeat ack is received (a bare TCP connect does
  not count as "reconnected" for backoff-reset purposes -- guards against a
  hub that accepts connections but cannot actually serve them).

## 3. Envelope format

One envelope shape for everything that crosses the spoke<->hub boundary,
whichever direction. A single shape (rather than N distinct message types)
keeps routing, at-least-once bookkeeping, and audit logging uniform;
`kind` + `payload` carry the semantic differences.

```jsonc
{
  "envelope_id": "01J...",        // ULID, generated by the ORIGINATOR (spoke
                                    // for spoke-authored envelopes, hub for
                                    // hub-authored ones e.g. presence pings).
                                    // The idempotency key for at-least-once
                                    // dedup (section 5).
  "workspace_id": "ws_...",        // Hard boundary. Must match the sender's
                                    // JWT workspace_id claim; hub rejects
                                    // (400) any mismatch, does not silently
                                    // correct it.
  "kind": "execute_command.request", // See section 3.1 for the enum.
  "from": {
    "machine_id": "mach_...",
    "member_id": "mem_..." | null  // null for machine-level envelopes
                                    // (heartbeat, presence); required for
                                    // member-attributed ones.
  },
  "to": {
    "machine_id": "mach_..." | null, // null when the hub itself is the
                                       // recipient (e.g. presence announce);
                                       // set when relaying to another spoke.
    "member_id": "mem_..." | null
  },
  "ts": "2026-07-04T00:00:00.000Z", // Originator's wall clock, informational
                                     // only -- never used for ordering or
                                     // TTL math (see section 6, clock skew).
  "ttl_ms": 30000 | null,           // Optional. How long the hub may hold
                                     // this envelope undelivered before
                                     // giving up (section 6). null means
                                     // "use the kind's default TTL"
                                     // (section 3.1).
  "correlation_id": "01J..." | null, // Set on a response/result envelope to
                                       // the envelope_id of the request it
                                       // answers. Lets a caller match a
                                       // fire-and-forget submission to its
                                       // eventual result without blocking
                                       // the HTTP request that submitted it.
  "payload": { /* kind-specific body */ }
}
```

Envelopes are submitted and delivered as JSON. A future binary framing
(e.g. for high-volume file-transfer chunks) is out of scope for this
design and would use a distinct `kind` family with a documented departure
from this shape, not a silent variant.

### 3.1 `kind` enum and default TTLs

| kind | direction | default ttl_ms | notes |
|---|---|---|---|
| `presence.announce` | spoke -> hub | n/a (not queued) | Sent on connect and on every member registration/deregistration. Full snapshot of the spoke's currently-attached members, not a diff (see section 4). |
| `presence.heartbeat` | spoke -> hub | n/a (not queued) | Cheap liveness ping, no member snapshot. |
| `presence.ack` | hub -> spoke | n/a | Acks a heartbeat/announce; carries the hub's view of `next_heartbeat_due_ms` (section 4). |
| `execute_command.request` | spoke -> hub -> spoke | 30000 | Relayed command dispatch (master plan section 7.2). |
| `execute_command.result` | spoke -> hub -> spoke | 60000 | Longer TTL than the request: a slow command may finish after the caller's own request TTL expired the queued copy of the REQUEST, but the result should still get a chance to reach a caller who reconnects. |
| `execute_command.long_running_update` | spoke -> hub -> spoke | 60000 | Progress/spill updates for long_running tasks (execute-command.ts long_running wrapper), one envelope per update, `correlation_id` = the originating request's envelope_id. |
| `send_message.deliver` | spoke -> hub -> spoke | 15000 | Interactive prompt injection payload (mode (b), master plan section 8). Short TTL: injecting a stale prompt into a live session is worse than dropping it. |
| `send_message.ack` | spoke -> hub -> spoke | 15000 | Delivery/read acknowledgement, `correlation_id` set. |
| `execute_prompt.request` / `.result` | spoke -> hub -> spoke | 30000 / 120000 | Mode (a) one-shot spawn relayed to the target machine; result TTL is long because headless CLI invocations can run minutes. |
| `event.broadcast` | spoke -> hub -> spoke(s) | 5000 | Fleet event-bus fan-out (`src/services/event-bus.ts`) scoped to workspace, fanned to every OTHER connected spoke in the same workspace. Best-effort; not queued past TTL (a missed event does not need catch-up, since polling tools re-derive current state). |
| `revocation.notice` | hub -> spoke | n/a (not queued; re-sent on next connect until acked) | Hub tells a spoke one of its cached JWTs/keys is revoked; spoke must drop the session. Special-cased: never silently dropped even past a nominal TTL, because a missed revocation is a security issue, not a UX one. Re-delivered on every reconnect until the spoke acks it. |

Kinds not listed (file-transfer chunks, dashboard-facing REST calls) are
out of scope for this envelope wire format; the dashboard talks to the hub
over its own REST+SSE API (master plan Addendum item 7), not this
spoke-facing envelope channel.

### 3.2 Envelope size limit

64 KiB per envelope (soft cap enforced by the hub, HTTP 413 above it). Large
payloads (file transfer, big command output) MUST be chunked at the
application layer (kind-specific: e.g. `execute_command.result` carries a
pointer/handle to a separately-fetched blob once output exceeds the cap,
matching the existing local "output spill" behavior in
`src/tools/execute-command.ts`) rather than growing the envelope. This
keeps the hot relay path (presence, small command results, prompt
injection) latency-predictable regardless of what a small minority of
large payloads are doing.

## 4. Presence and heartbeat semantics

- **Connect**: spoke opens the SSE stream, then POSTs `presence.announce`
  with the full list of currently-attached members
  `{member_id, provider, work_folder, status: "online"|"busy"}[]`. The hub
  replaces its entire prior view of that machine's members with this
  snapshot (not a merge) -- this makes reconnect-after-crash
  self-correcting: whatever the exe reports NOW is truth, no stale entries
  can survive a restart.
- **Heartbeat interval**: 15s, spoke-initiated. The hub's presence-ack
  response carries `next_heartbeat_due_ms` (server-authoritative, defaults
  to 20s = 15s interval + 5s grace) so a slow/overloaded spoke can be told
  to back off without a protocol change, and so the hub's own timeout logic
  and the spoke's send cadence are provably consistent (the spoke does not
  need to independently guess the hub's timeout).
- **Hub-side timeout**: a spoke is marked OFFLINE after missing 2
  consecutive expected heartbeats (i.e., ~2x `next_heartbeat_due_ms`,
  default ~40s) with no traffic of any kind from that spoke (any envelope
  counts as implicit liveness, not just heartbeats -- a busy spoke mid
  large-file relay should not be falsely marked offline just because it
  hasn't sent a dedicated heartbeat frame recently).
- **Member-level status** (online/busy, per apra-fleet-2xs.7's local state
  machine) is carried as a field inside `presence.announce`/incremental
  status-change envelopes (a lighter-weight `presence.member_status`
  sub-kind may be added when 2xs.7 lands; this document reserves the name
  but does not spec its body, since 2xs.7 is tier-2-local and unimplemented
  as of this writing) -- the hub relays it upward for dashboard display and
  downward to OTHER spokes in the workspace that need to know (e.g. a
  caller deciding mode (a) vs (b) for a remote execute_prompt must know the
  TARGET's status, which lives on the target's machine -- this is why
  presence must be workspace-fanned, not hub-private).
- **Disconnect handling**: on detecting a stream drop (read error, or the
  heartbeat timeout above), the hub marks the machine and all its members
  OFFLINE, timestamps `offline_since`, and begins offline-queueing per
  section 6. It does NOT immediately purge the machine/member registry
  rows (those are durable CRUD data, not presence) -- only the volatile
  presence layer changes.
- **Reconnect**: spoke reconnects (fresh SSE stream + fresh
  `presence.announce`), hub flips the machine back ONLINE, and immediately
  begins flushing anything queued for it (section 6) in `envelope_id`
  order per target member (FIFO per member, not globally -- see section 5).

## 5. Relay routing and at-least-once delivery

1. A spoke POSTs an envelope addressed to `to.machine_id` (possibly its own
   machine, for intra-machine relay is not a real case since tier-1<->tier-2
   handles that locally -- `to.machine_id` is always a DIFFERENT machine in
   practice, but the hub does not special-case same-machine-id; it would
   just route back to the same connection).
2. Hub validates: JWT workspace_id matches envelope workspace_id; if `to`
   is set, the target machine_id/member_id must resolve to a member CRUD
   record in the SAME workspace_id (cross-workspace addressing is rejected
   with 403 at admission, never silently dropped -- a workspace-boundary
   violation is loud, not quiet).
3. Hub assigns the envelope a durable position in a per-target-member
   ordered queue (implementation is the ADR's concern; the wire contract
   only requires FIFO-per-target-member ordering and that a delivered
   envelope is not removed from the queue until acknowledged -- see below).
4. If the target machine's spoke channel is ONLINE: the hub pushes the
   envelope down the SSE stream immediately. The spoke's tier-2 code
   processes it and returns an HTTP 202 style ack via a lightweight
   `POST /v1/ack {envelope_id}` call (a plain ack, not itself an envelope,
   to keep acking cheap and not recursively subject to its own delivery
   guarantees). Only on receiving this ack does the hub retire the envelope
   from the target's queue.
5. If the ack is not received within `ack_timeout_ms` (default 10s -- short,
   because the common case is a live connection where ack latency should be
   well under a second; a slow ack usually means the spoke silently died
   between SSE-push and ack, not that the work is slow), the hub
   RE-DELIVERS the same envelope (same `envelope_id`) on the next
   opportunity (immediately if still connected, or on next reconnect if
   not). This makes delivery at-least-once, not exactly-once: **spoke-side
   handlers MUST be idempotent per `envelope_id`** -- tier 2 keeps a
   short-lived (5 minute) seen-envelope_id cache and no-ops a duplicate
   delivery rather than re-executing an already-completed
   `execute_command.request`, for example.
6. Ordering guarantee: FIFO is scoped to (workspace_id, target member_id)
   pairs only. Two envelopes to different members, or from different
   senders to the same member, have no cross-pair ordering guarantee beyond
   "queued position at admission time" -- this matches what a real
   multi-sender system can honestly promise and avoids a global sequencer
   bottleneck.
7. Fan-out kinds (`event.broadcast`) are the one exception to
   per-target-member queueing: the hub expands them at admission time into
   one queued copy per currently-known member in the workspace (excluding
   the sender), each independently subject to sections 5-6. A member that
   is offline at fan-out time and reconnects after the event's TTL simply
   never receives that particular event -- acceptable per section 3.1's
   "no catch-up needed" note.

## 6. Offline-spoke queueing and TTL semantics

- Every relay-kind envelope (everything in section 3.1 except the
  `presence.*` and `revocation.notice` special cases) is queued at
  admission regardless of whether the target is currently online -- the
  online/immediate-push path in section 5 step 4 is simply "deliver
  attempt #1 happens with ~zero queue residency", not a different code
  path. This means there is exactly one delivery mechanism to reason about,
  not two (no separate "online fast path" vs "offline slow path" whose
  behaviors could silently diverge).
- **TTL is measured from admission (hub-clock), not from the originator's
  `ts` field** -- the wire format's `ts` is informational/audit-only
  (section 3, clock skew across machines is not something this protocol
  corrects for). The hub stamps its own admission time and expires
  strictly off of that.
- On TTL expiry with the envelope still undelivered/unacked: the hub
  generates a synthetic `<kind>.failed` result envelope back to the
  ORIGINATING spoke/member (e.g. an expired `execute_command.request`
  yields an `execute_command.result` with `payload.status: "target_offline_
  ttl_expired"`), so a caller waiting on `correlation_id` gets a definite
  answer rather than silence. This applies uniformly to relay kinds; it
  does NOT apply to `presence.*`/`revocation.notice`, which are not
  queued/TTL'd in the normal sense (see their table rows).
- **Per-member queue depth cap**: 1000 envelopes or 8 MiB, whichever is hit
  first (defensive default, tunable per workspace tier later). Exceeding
  the cap rejects the NEWEST admission attempt with an explicit "target
  queue full" error to the sender (never silently drops an OLDER queued
  item to make room -- silently dropping older, possibly
  already-in-flight-looking work is worse than a loud, immediate failure of
  the newest one).
- **Revocation is never TTL-dropped** (section 3.1): `revocation.notice`
  is re-announced on every spoke reconnect attempt until acked, independent
  of any queue/TTL bookkeeping, because letting a revoked credential keep
  working past a missed notice is a security regression, not a UX
  inconvenience.
- **Long-offline spokes**: a machine offline for more than 24h has its
  queued envelopes (other than `revocation.notice`, which persists
  indefinitely until acked or the credential is deleted outright) hard-
  expired and synthetic-failed per the TTL path above, even though
  individual envelope TTLs are much shorter -- this bound exists purely so
  the hub's storage for a permanently-decommissioned machine does not grow
  unbounded; it is a backstop, not the normal expiry path (which is almost
  always the per-kind TTL in section 3.1, expected to fire within seconds
  to minutes).

## 7. What this design explicitly defers

- The concrete server-side storage for the per-target-member ordered queue
  (Redis Streams + consumer groups vs Postgres-table-as-queue vs NATS
  JetStream) is the persistence ADR's decision (apra-fleet-us9.3), not
  this document's. Section 5's requirements (FIFO-per-target-member,
  ack-then-retire, at-least-once) are the acceptance criteria the chosen
  backend must satisfy, written so the ADR can be graded against them.
- Member-level status sub-kind body (`presence.member_status`) is a name
  reservation only, pending apra-fleet-2xs.7 (tier-2-local state machine)
  landing; do not implement its wire body from this document alone --
  confirm against 2xs.7's actual state names first.
- Binary/chunked framing for large payloads (file transfer, oversized
  command output) is named as an escape hatch (section 3.2) but not
  specified; it is in scope for apra-fleet-us9.12 (hub-brokered file
  transfer).
- Backpressure/rate-limiting per workspace (protecting the hub from one
  noisy tenant) is not addressed here; it is an operational concern for the
  hub service MVP (apra-fleet-us9.4), layered on top of this wire contract
  without changing it (e.g. via HTTP 429 on admission, which this document
  already treats as a normal rejectable-at-admission outcome per section 5
  step 2's "reject at admission" pattern).
