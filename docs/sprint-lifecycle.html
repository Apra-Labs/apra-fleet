Life of a Sprint -- Planning to Harvesting

The full loop, annotated with real sprint traces (kb-trust-ops + kb-inflight-capture, 2026-07-07). Updated: learnings are now captured AT discovery time in Plan and Execute; the KB Agent curates at harvest.

Plan

PLANNER -- strongest model, fresh session

kb_statscoverage on the plan's key symbols -> quantitative model assignment: coverage >= 0.8 -> cheap model, < 0.3 -> premium + front-load; the number is cited in PLAN.md.

Writes PLAN.md: phased tasks, riskiest first, exact model per task citing KB coverage, VERIFY closing each phase. Commits as pm-planner.

3

Plan Review (loop to APPROVED)

PLAN-REVIEWER -- strongest model, fresh session
Checks coverage, design compliance, risk order, self-containedness -- and attacks the plan: verifies factual anchors against source, war-games the security design.
kb_captureplan-review discoveries (a design flaw class, an attack path, a corrected factual anchor) are captured at discovery time too -- they no longer evaporate into feedback.md history.
CHANGES NEEDED -> PM updates design.md if the design itself was wrong -> planner revises -> re-review. APPROVED -> beads tasks + progress.json generated.
planner <-> plan-reviewer loop until APPROVED -- the cheapest place to catch a flaw

4

Execute (per phase)

DOER -- the model the planner assigned, per task streak
kb_session_prime+kb_queryretrieve-first: a CONFIRMED hit skips the source read entirely.
code_context / code_impact / code_testsstructural navigation; code_context inlines matching KB entries; code_tests finds targeted tests to run.
kb_captureCAPTURE AT DISCOVERY TIME: coding standards, conventions, constraints discovered while working are captured the moment they are found (clamped to INFERRED, tagged sprint/phase). Trust rule: CONFIRMED fully; INFERRED is a strong hint -- verify against source when correctness matters.
kb_feedbackif a retrieved entry proves wrong in practice -> flag + stale (downvote), never silently ignore.
Implement -> fast tests -> commit as pm-doer -> update progress.json -> next task. Fail-then-pass tests required for behavior changes.
VERIFY checkpoint ends the phase: build + full suite + gitnexus analyze (incremental re-index so later phases see new symbols) + push. Then STOP for review.

5

Code Review (per phase)

REVIEWER -- strongest model, fresh session, independent
Reads diff + PLAN + requirements + design; runs build and tests itself; usescode_impactfor blast radius;kb_queryto skip re-derivation.
Security-critical work gets live attacks, not test-reading: the reviewer executes the exploit against compiled code and confirms it fails.
kb_captureREVIEWERS CAPTURE TOO: flaw classes, attack paths, constraints the diff exposed are captured at discovery (INFERRED, tagged) -- both the code reviewer AND the plan-reviewer (stage 3). Review insights no longer live only in feedback.md history.
Writes feedback.md verdict as pm-reviewer. CHANGES NEEDED -> doer fixes -> re-review (max 3 cycles). The verdict is also the knowledge quality gate.
doer <-> reviewer loop until APPROVED, then the phase's knowledge is harvestable

6

Harvest = Curate (after EVERY verdict)

KB AGENT -- cheap model, dispatched by PM automatically; CURATOR, not sole capturer
kb_query tag:phasegather the phase's in-flight captures (everything the planner/doer/reviewer captured at discovery time, found by sprint/phase tag).
kb_promotecurate each against the verdict: validated by APPROVED code -> promoted to CONFIRMED with evidence; wrong -> kb_feedback (flag + stale); fine-but-unvalidated -> left INFERRED. Promotion remains the ONLY path to CONFIRMED.
kb_captureresidual gaps the in-flight captures missed (the curator's fallback job). Corrections supersede for real; contradictions get flagged.
kb_exportwrites .fleet/kb-canonical.json and AUTO-COMMITS it (pm-kb identity, pathspec-only, non-fatal) -- the learning enters the codebase with zero manual steps. Fleet-mode sessions also get the transcript harvest autowire (UNVERIFIED tier).

8

Next sprint starts smarter

THE COMPOUNDING LOOP
The next planner primes and inherits everything: live CONFIRMED entries first, graph-expanded neighbors below, project + global bibles seeding any cold start. Knowledge captured once is retrieved forever; a correction propagates forever; your approved directives apply forever.
MEASURED ACROSS THE 4 SPRINTS OF 2026-07-06/07 Retrieval hit rate grew to 76% (235 reads); planners went from cold-KB guesswork to citing entry IDs per task; each sprint's plan was measurably cheaper to produce than the last because exploration was replaced by retrieval.
The one-sentence version: a sprint is a loop of gated handoffs -- requirements (human intent) -> plan (KB-informed, capturing discoveries as it explores) -> execute (retrieve-first, capturing at discovery time, fail-then-pass) -> review (independent, sometimes literally attacking the code) -> harvest (the curator: verdict-gated promotion of what the phases already captured, self-committing bible) -- learning happens WHERE the discovery happens, trust is assigned where the verdict happens, and every later phase and sprint starts better-informed than the one before it.
